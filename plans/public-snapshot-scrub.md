# Plan: Public snapshot scrub

## Scope

A script that produces the **publicly clonable, anonymized snapshot** of the data repo for contributor onboarding. Walks the public gitsheets repo, applies pseudonymization rules, writes a clean snapshot to a separate public repo (`codeforphilly-data-snapshot`).

Out of scope: any private-store handling (the private store doesn't get a public snapshot — devs use empty or fixture-seeded private storage); the cutover-time orchestration (that's [`cutover-prep`](cutover-prep.md)). This plan delivers the *script*; how it gets invoked in CI is a deploy concern.

## Implements

- [behaviors/storage.md](../specs/behaviors/storage.md) — the "Dev-environment data" section (scrubbed snapshot conventions)
- [behaviors/private-storage.md](../specs/behaviors/private-storage.md) — confirms no private-store data leaks into the snapshot (trivially true; the script only reads the public repo)

## Approach

### Script entry point

`apps/api/scripts/scrub-data.ts`:

```bash
npm run -w apps/api script:scrub-data -- \
  --source=./codeforphilly-data \
  --target=./codeforphilly-data-snapshot \
  [--seed=2026-05-15] [--dry-run]
```

`--seed` makes the pseudonymization deterministic. Same seed = same scrubbed output. Useful for reproducible CI snapshots.

### What gets scrubbed

The public repo by design has no real PII fields — emails live in the private store, not here. But it does have:

- **Commit authors** carrying `<slug>@users.noreply.codeforphilly.org` — pseudonymous already by [behaviors/storage.md](../specs/behaviors/storage.md), but the *slug* itself is a real person's username. Scrub for the public snapshot: pseudonymize slugs.
- **Person.fullName / firstName / lastName** — real names. Replace with faker-generated names (keyed off the seed so the same person always gets the same pseudonym across snapshots).
- **Person.bio** — user-authored prose; could leak identifying info. Replace with a lorem-ipsum-shape paragraph in the snapshot.
- **Person.slackHandle** — replace with a pseudonym matching the new slug.
- **Person.githubLogin / githubUserId** — clear these.
- **Person.slackSamlNameId** — clear (this is for SAML continuity; doesn't apply to snapshot).
- **Person.avatarKey** — clear (no attachment data follows along; UI falls back to generic-avatar).
- **Project fields** — keep as-is. Project READMEs, summaries, etc. are public-by-design.
- **Project.maintainerId / membership personIds** — re-point to the pseudonymized Persons (the IDs are the same UUIDs internally; we don't re-issue UUIDs, just rename the human-facing fields).

### Single squashed commit

The snapshot repo is **a single squashed commit per snapshot run** — no history. Reasons:

- History in the snapshot would expose pseudonym → real-name patterns over time
- Snapshot consumers (contributors) don't need history; they want a starting state
- Smaller clone, faster onboarding

Each snapshot is a separate orphan branch + tag: `snapshot-2026-Q2-scrubbed`, `snapshot-2026-Q3-scrubbed`, etc. Tag is the durable handle.

### Slug pseudonymization

The hard one. Slugs appear in URLs, in commit messages historically, in record cross-references. Strategy:

1. Build a `realSlug → pseudoSlug` map at start, deterministically from the seed:
   - `realSlug` → `hash(seed, realSlug) % wordlist[N]` → produces e.g. `friendly-armadillo-12`
2. For every record, replace any field that holds a slug:
   - `Person.slug`
   - `Project.chatChannel` (sometimes contains a name; leave the channel name itself but verify it doesn't contain PII; safest: replace with `chat-<projectSlug>`)
   - `ProjectBuzz.slug` (derived from headline; might leak; replace with `<projectSlug>-buzz-<sequence>`)
   - File paths (since records are keyed by slug in the path template)
3. Inside record bodies, replace slug references (`Person.fullName` is already replaced; `Project.overview` may contain `@slug` mentions — replace those too)
4. Slugs that don't have a pseudonym mapping (e.g., post-snapshot signups in a hypothetical future) are kept as-is

### Faker seeding

Use `@faker-js/faker` with `faker.seed(seedNumber)` to make name generation deterministic. Same seed = same pseudonyms.

### Commit author identity in the snapshot

The single squashed commit author is `Code for Philly <snapshot@users.noreply.codeforphilly.org>`. Commit message:

```
snapshot: anonymized data export from <timestamp>

Generated by apps/api/scripts/scrub-data.ts.
Source revision: <hash of source repo's HEAD>
Seed: <pseudonymization seed>

This snapshot is intended for contributor onboarding. It contains no
real names, no emails, no GitHub identities. All personal data has
been pseudonymized via the documented scrubbing rules.

See apps/api/scripts/scrub-data.ts and specs/behaviors/storage.md.
```

### Verification step

After producing the snapshot, the script runs a verification pass:

- Grep for `@` in record files → should be zero hits (no emails)
- Grep for any real slug from the source repo → should be zero hits in scrubbed records
- Grep for `githubUserId`, `githubLogin`, `slackSamlNameId` → should be zero or null in every Person record
- Diff record counts (people, projects, etc.) between source and snapshot → should match

If verification fails, the script exits non-zero and doesn't write the output.

### How it's invoked in production

Out of scope for this plan, but documented for [`cutover-prep`](cutover-prep.md):

- A scheduled GitHub Actions workflow in the data repo runs the script weekly
- Pushes the resulting snapshot to `codeforphilly-data-snapshot` (force-push to the snapshot branch since each run is a squashed commit)
- Tags the run with the date

### Local dev usage

A contributor can run this against any local data repo to produce their own snapshot:

```bash
npm run -w apps/api script:scrub-data -- --source=../codeforphilly-data --target=./scratch/my-snapshot
```

Useful for testing the script itself and for sharing reproducible test data.

## Validation

- [ ] Run against a fixture data repo → produces a snapshot with no real names, no emails, no GitHub identities, no Slack handles
- [ ] Re-run with the same seed → byte-identical output (verify by hashing the target tree)
- [ ] Run with different seeds → different pseudonyms; same record structure
- [ ] Snapshot's git history is a single commit
- [ ] Verification pass catches a deliberately-leaked email (test by injecting `email = "real@example.com"` into a fixture record and confirming the script exits non-zero)
- [ ] Slug pseudonymization is reversible-via-the-map but the map itself isn't published with the snapshot
- [ ] Project record counts match source vs. snapshot
- [ ] Snapshot clones + boots a fresh dev API (`STORAGE_BACKEND=filesystem`, empty private storage)

## Risks / unknowns

- **Faker wordlist exhaustion.** At 1240 users, a 2-word `<adjective>-<animal>` namespace might have enough entropy with a numeric suffix. Confirm during implementation; bump to 3-word if needed.
- **Slug references inside markdown.** A project overview that says "...thanks to @janedoe..." needs the mention rewritten. The replacement is straightforward (regex scan + map lookup); the cost is making sure we don't miss any embedded reference.
- **Slug uniqueness in the pseudonym space.** Hash collisions handled by appending a numeric suffix. Confirm the wordlist + suffix produces stable unique-per-seed pseudonyms.
